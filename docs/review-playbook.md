# Review Playbook — ahillary-will-do-abs

**Версия:** 3.0
**Дата:** 2026-06-25

Единственный источник процедуры по карточке — и для автопилота, и для ручной работы.
Проект: личный локальный трекер (FastAPI + SQLite + React/Vite/Tailwind).
Автопилот перечитывает этот файл на КАЖДОЙ карточке — правки подхватываются без перезапуска.

**Полный цикл разработки (никогда не короче, можно длиннее):**
0. Env-preflight
1. Прочитать карточку
2. **План** — изучить код, сформулировать ЧТО и КАК + собрать 5-7 UI-сценариев
3. Реализация + зелёные гейты качества
4. **Ревью** — `/code-review` + `ecc:security-review` → починить ВСЕ находки
5. **Playwright** — живьём пройти 5-7 сценариев ТОЛЬКО через UI (без url/curl) ↔ фиксы
6. Финализация (коммит) + финальные сценарии

Ни один шаг нельзя пропустить. Шаги 4 и 5 имеют свои циклы фиксов.

## Плейсхолдеры (подставляет оркестратор)
- `{{BASE_URL}}` — фронт (Vite), обычно `http://localhost:5173`
- `{{API_HEALTH_URL}}` — health бэкенда, обычно `http://localhost:8000/health`
- `{{CREDS}}` — вход в приложение: `SEED_USER_EMAIL` / `SEED_USER_PASSWORD` из `backend/.env`
- `{{BRANCH_NAME}}` — ветка `feat/…` или `fix/…`
- `{{TASK}}` / `{{ACCEPTANCE}}` — заголовок и блок «Критерии приёмки» карточки

## Карта флоу

```
ШАГ 0  Env-preflight
   ▼
ШАГ 1  Прочитать карточку (2 абзаца, TASK_SURFACE)
   ▼
ШАГ 2  ПЛАН: изучить код → ЧТО и КАК → 5-7 UI-сценариев
   ▼
ШАГ 3  Реализация (ветка от main, TDD, минимальный диф) → гейты зелёные
   ▼
ШАГ 4  РЕВЬЮ: /code-review + ecc:security-review ──┐  есть находки → чинить ВСЕ → перегейт
   ▼ (находок нет)                                 └──────────────────────────────┘
ШАГ 5  PLAYWRIGHT: живьём 5-7 сценариев через UI ─┐  FAIL/баги → ШАГ 5а фиксы (≤3 цикла)
   ▼ (всё PASS)                                    └──────────────────────────────┘
ШАГ 6  Финализация: коммит на ветке + финальные сценарии
```

**Условие финала:** гейты зелёные **И** ревью без находок **И** все сценарии Playwright = PASS.
Любой провал на 4/5 → цикл фиксов; после исчерпания циклов → `AUTOPILOT_RESULT: FAILED`.

---

## ШАГ 0 — Env-preflight

**Цель:** убедиться в состоянии окружения до работы.

1. Дерево чистое, ты на свежем `main` (`git status`, `git pull --ff-only`).
2. Если карточка наблюдаема через UI — подними/проверь серверы: `curl -s {{API_HEALTH_URL}}` → `{"status":"ok"}`; фронт `{{BASE_URL}}` → 200. *(Это единственное допустимое использование curl — preflight окружения, НЕ проверка фичи.)*
3. Playwright MCP жив (браузер отвечает). Если нет — это блокер для Шага 5, подними заранее.

```
ENV-PREFLIGHT:
- git: OK (main, clean)
- api_health: OK | FAIL — <ответ>
- frontend: OK | FAIL | N/A
- playwright: OK | FAIL
ВЕРДИКТ: READY | NOT_READY
```
При `NOT_READY` — перечисли блокеры и не продолжай.

---

## ШАГ 1 — Прочитать карточку

1. Прочитай карточку целиком: заголовок, описание, чеклист **«Критерии приёмки»** = Definition of Done.
2. Объясни задачу в РОВНО 2 абзацах: о чём задача, что добавляет/улучшает, как влияет на проект.
3. Определи **TASK_SURFACE**: `backend-only` | `backend+UI` | `UI-only`. Смотри на код, а не на заголовок. Даже у «бэкенд» карточек результат почти всегда проявляется в UI (новое поле в форме, изоляция данных по юзеру и т.п.) — найди эту поверхность, её и будешь гонять на Шаге 5.

---

## ШАГ 2 — План (изучить код и сформулировать подход)

**Цель:** до единой правки понять ЧТО и КАК делать — как в plan mode.

1. **Изучи существующий код** вокруг задачи: модели/роуты/компоненты/хелперы, которые затронешь или переиспользуешь. Не дублируй то, что уже есть.
2. Сверь формулировку карточки с кодом: если она называет компонент/поле/маршрут, которого нет или который значит другое — зафиксируй расхождение; если оно меняет ЧТО строить — выбери разумный дефолт и задокументируй (не спрашивай, автономный режим).
3. Сформулируй **план**: список файлов к изменению, новые сущности/миграции, точки переиспользования, риски. Кратко, но конкретно (это твой внутренний plan mode — без ожидания подтверждения человека).
4. Составь **5-7 пошаговых UI-сценариев** проверки (они же станут финальными на Шаге 6).

**ЖЁСТКИЕ ПРАВИЛА СЦЕНАРИЕВ:**
- Только через интерфейс. Запрещено: прямые URL вместо кликов, прямой SQL, curl, обход UI.
  - ПЛОХО: «Перейдите на `/workouts`».
  - ХОРОШО: «Откройте {{BASE_URL}}, войдите как {{CREDS}}, нажмите "Тренировки" в меню».
- До каждого экрана — реальный кликабельный путь. Нет пути → экран недостижим через UI, выноси в «Известные ограничения».
- Каждый шаг = конкретное действие + конкретный видимый результат.
- Ярлыки/заголовки бери из реально отрендеренного UI (сверься с компонентами); не выдумывай.
- Привяжи каждый сценарий к критерию приёмки. Покрой happy-path, граничные случаи и хотя бы один негативный/ошибочный путь.

**Формат сценариев:**
```
### S1 — <название>
- Цель: <что проверяем>
- Предусловие: <если нужно>
- Шаги:
  1. <действие> → <ожидаемый видимый результат>
  2. …
- Итог: <одна строка>
- Закрывает: <критерий ACCEPTANCE>
```

## ШАГ 3 — Реализация

**Цель:** реализовать ровно критерии приёмки.

- Ветка `{{BRANCH_NAME}}` от свежего `main`. На `main` не коммить.
- TDD там, где даёт сигнал: бэкенд-логику покрывай тестами (RED → GREEN); трудно-тестируемый UI гейтируй `tsc --noEmit` + живой Шаг 5.
- YAGNI: ровно критерии приёмки. Минимальный диф: НЕ давай ruff/prettier/eslint трогать несвязанные строки.
- Малые связные файлы (≤800 строк), явная обработка ошибок на границах, секреты только из `.env`.
- Для UI-карточек сделай результат **наблюдаемым** в браузере — иначе Шаг 5 нечем гонять.
- Не ломай существующие тесты и общие хелперы. Миграции SQLite — только через `op.batch_alter_table`.

**Гейты качества (прогоняй до зелёного; повторяй после любых правок на Шагах 4-5):**
- Backend: `cd backend && ruff check . && ruff format --check . && pytest -q`
- Frontend: `cd frontend && npx tsc --noEmit && npx eslint src --max-warnings 0` (+ `npx vitest run`, если есть тесты по карточке)
- Модель-меняющая карточка обязана содержать миграцию Alembic; прогони `alembic upgrade head && alembic downgrade -1 && alembic upgrade head`.

До **3 циклов** фиксов на гейтах; после 3 — стоп и `AUTOPILOT_RESULT: FAILED`.

## ШАГ 4 — Ревью кода и безопасности (обязательно)

**Цель:** прогнать написанный код через ревью-скиллы и устранить ВСЕ находки до проверки в UI.

**Процедура:**
1. На незакоммиченном дифе ветки запусти **оба** скилла:
   - `/code-review` (локальный, по текущим изменениям — НЕ облачный `ultra`).
   - `ecc:security-review`.
2. Собери все находки обоих скиллов в единый список с severity.
3. **Почини АБСОЛЮТНО ВСЕ находки** — любого severity (CRITICAL/HIGH/MEDIUM/LOW/INFO). «Потом» не бывает.
   - Если находка — доказуемо ложное срабатывание или намеренно вне scope карточки: НЕ молчи, а письменно обоснуй почему (одна строка на находку). Только тогда она не считается. По умолчанию — чинить.
4. После фиксов **перепрогони гейты качества (Шаг 3)** и **оба ревью-скилла снова**.
5. Повторяй, пока оба скилла не вернут пустой actionable-список. **Максимум 3 цикла**; если после 3 циклов остаются настоящие блокеры — стоп и `AUTOPILOT_RESULT: FAILED` со списком оставшегося.

**Формат вывода Шага 4:**
```
## РЕВЬЮ
code-review: <N находок> | security-review: <M находок>
- R-01: severity=<...> | источник=code-review|security | файл:строка | суть | действие=исправлено|обосновано-вне-scope
- …
ИТОГ РЕВЬЮ: CLEAN | FAILED (<k> неустранённых)
```

## ШАГ 5 — Playwright: живой прогон 5-7 сценариев через UI

**Обязательно для любой карточки с наблюдаемой поверхностью** (а такая есть почти всегда — см. Шаг 1.3). Никаких curl/HTTP/URL-проверок вместо UI. Если карточка совсем без UI-поверхности (например, внутренний рефактор без видимого эффекта) — явно обоснуй это в отчёте и прогоняй её через UI там, где её эффект всё-таки проявляется (фича-потребитель).

**Прогони ВСЕ 5-7 сценариев из Шага 2 через MCP Playwright на `{{BASE_URL}}`.** Ты сам кликаешь по живому UI — не пересказывай код, а воспроизводи действия.

**Процедура для каждого сценария:**
1. Иди ровно по шагам: каждое действие — реальный клик/ввод через Playwright. `browser_navigate` — ТОЛЬКО на стартовый `{{BASE_URL}}` или `/login`; дальше только клики, без прямых URL и без curl.
2. После каждого шага — snapshot, сверяй фактический экран с ожидаемым ПОБУКВЕННО (ярлыки, заголовки, тексты).
3. Снимай `browser_console_messages` и `browser_network_requests`.
4. НИКОГДА не вызывай `browser_close` — это рвёт мост к Chrome пользователя.

**ЧЕК-ЛИСТ КРИТЕРИЕВ ПРОВАЛА (любой пункт = баг, строгость не занижать):**
- `[НЕТ_КНОПКИ]` — требуемого сценарием элемента нет на экране.
- `[ЯРЛЫК≠СЦЕНАРИЙ]` — элемент есть, но текст не совпадает со сценарием.
- `[ЛАГ]` — зависание/спиннер дольше ~2с, дёрганый редирект.
- `[КОНТРОЛ_НЕ_ОТРАЖАЕТ_РЕЗУЛЬТАТ]` — после мутирующего действия его контрол не изменился (не исчез / не задизейблился / не сменил ярлык). Тост рядом НЕ засчитывается.
- `[КОНСОЛЬ]` — любой `console.error` / необъяснённый warning.
- `[NETWORK]` — любой запрос 4xx/5xx (кроме явно ожидаемых сценарием отказов).
- `[НЕТ_КЛИК_ПУТИ]` — до экрана нет кликабельного пути; ввод URL руками запрещён.
- `[НЕОЧЕВИДНЫЙ_ПУТЬ]` — чтобы выполнить шаг, нужно угадывать/скроллить к неочевидному месту.
- `[ENV]` — `{{API_HEALTH_URL}}` не зелёный / серверы упали → вернись на Шаг 0.

**ПРАВИЛО СТРОГОСТИ:** PASS по сценарию — только при 100% совпадении всех шагов и отсутствии любого пункта чек-листа. Малейшее сомнение → это баг.

**Формат вывода Шага 5:**
```
## ПРОГОН
| Сценарий | Вердикт |
| S1       | PASS / FAIL |
| …        |             |
ИТОГ: PASS (0 багов) | FAIL (<N> багов)

## BUG_LIST
- BUG-01: сценарий/шаг S<k>.<n> | тип [...] | severity | факт | ожидание | репро
(нет багов → "## BUG_LIST\n(пусто)")
```

### Шаг 5а — Фиксы → возврат на Шаг 5
**Когда:** `ИТОГ: FAIL` / непустой BUG_LIST.
- Класс бага: `КОД_ФИЧИ` | `СЦЕНАРИЙ` (`[ЯРЛЫК≠СЦЕНАРИЙ]` — обычно правка сценария под реальный UI) | `ОКРУЖЕНИЕ` (`[ENV]` → Шаг 0).
- Каждый фикс проверяй повторным прогоном, а не рассуждением. Минимальный диф.
- После фиксов перепрогони гейты (Шаг 3). Если правка существенная — перепрогони и ревью (Шаг 4).
- **Максимум 3 цикла 5↔5а**; после 3 — стоп и `AUTOPILOT_RESULT: FAILED`.

---

## ШАГ 6 — Финализация + финальные сценарии

**Когда:** гейты зелёные + ревью CLEAN + все сценарии PASS.

### 6а. Финализация
- Закоммить логическими единицами на ветке (feat/fix/docs/test). НЕ пушить и PR не открывать (если оркестратор не задал иное в «Адаптации»).

### 6б. Финальные сценарии тестировщику
Пересобери финальную версию 5-7 сценариев (уже проверены на Шаге 5):
- Те же UI-only правила (без URL/SQL/curl), маршрут кликами от `{{BASE_URL}}`.
- Ярлыки/заголовки — точно как в реальном UI (после сверки на Шаге 5).
- Для каждого сценария: «Как срабатывает» (1-2 строки механики) + «Ожидаемый результат».
- В конце — матрица ожиданий (что / где / что должно быть видно) + «Известные ограничения».

## Правила (всегда)

- Где нужен выбор и в карточке нет указания — бери разумный дефолт, не спрашивай (автономный режим).
- «Баг» = поведение, расходящееся с критериями приёмки, или пункт из чек-листа Шага 5.
- Стек: backend — Python (venv) + FastAPI + SQLModel + SQLite; frontend — npm + Vite + React + TypeScript + Tailwind. НЕ вводи pnpm/yarn.
- Прод-сборку (`npm run build`) не гоняй — фронт проверяется `tsc --noEmit` + eslint + живым Playwright.
- В Playwright MCP НИКОГДА не вызывай `browser_close`. Проверка фичи — только через UI, не curl.
- **Dev-БД общая с пользователем — не засоряй её.** Для UI-сценариев сперва используй уже
  существующие/засиженные данные. Если для проверки ПРИШЛОСЬ создать сущности (вид спорта,
  упражнение, челлендж, ачивку, спонсора и т.п.) — давай им очевидный префикс `__test__` и
  **удали их через UI перед финализацией** (Шаг 6). Нельзя оставлять в базе спорты/записи,
  названные по коду карточки («B14 …», «F3 …», «smoke»). Тренировки/замеры/цели пользователя не трогай.
- На `main` не коммить.
```
