# Review Playbook — ahillary-will-do-abs

**Версия:** 1.0

Единственный источник процедуры по карточке — и для автопилота, и для ручной работы.
Проект: личный локальный трекер (FastAPI + SQLite + React/Vite/Tailwind), один пользователь.
Автопилот перечитывает этот файл на КАЖДОЙ карточке — правки подхватываются без перезапуска.

## Плейсхолдеры (подставляет оркестратор)
- `{{BASE_URL}}` — фронт (Vite), обычно `http://localhost:5173`
- `{{API_HEALTH_URL}}` — health бэкенда, обычно `http://localhost:8000/health`
- `{{CREDS}}` — вход в приложение: `SEED_USER_EMAIL` / `SEED_USER_PASSWORD` из `backend/.env`
- `{{BRANCH_NAME}}` — ветка `feat/…` или `fix/…`
- `{{TASK}}` / `{{ACCEPTANCE}}` — заголовок и блок «Критерии приёмки» карточки

## Шаг 0 — Префлайт
- Прочитай карточку целиком: заголовок, описание, чеклист **«Критерии приёмки»** — это и есть Definition of Done.
- Свериться с `~/.claude/CLAUDE.md` и правилами (ponytail: минимальное решение, immutable-данные, малые файлы, без лишних абстракций). Свериться со схемой/планом, если карточка их касается.
- Быть на свежем `main`, дерево чистое. Серверы нужны только если карточка наблюдаема через UI/HTTP — для скелета/моделей/парсеров не обязательно.

## Шаг 1 — Ветка
- Создай ветку `{{BRANCH_NAME}}` от `main`. На `main` не коммить.

## Шаг 2 — Реализация
- Реализуй ровно то, что в критериях приёмки, не больше (YAGNI). Следуй стеку и схеме данных проекта.
- Малые связные файлы (≤800 строк), явная обработка ошибок на границах, валидация внешнего ввода.
- Секреты только из `.env` (ключи Anthropic/Trello никогда не в коде). Файлы (скрины/видео) — на диск, в БД пути.
- Нетривиальная логика (парсер FatSecret, расчёт дефицита, PR-движок, правило закрытия ачивки) — оставь один прогоняемый тест (pytest, assert), как требует ponytail.

## Шаг 3 — Проверка (гейт качества)
Прогони применимое к карточке и добейся зелёного. До **3 циклов** фиксов; если после 3 всё ещё FAIL — стоп и FAILED.
- **Backend:** `ruff check`, `ruff format --check`, `pytest` (если есть тесты по карточке).
- **Frontend:** `tsc --noEmit`, eslint, `vitest` (если есть). Прод-сборку (`npm run build`) гонять НЕ нужно.
- **UI/HTTP-карточка:** при поднятых серверах — один смоук (`curl` на эндпоинт), а в режиме `--pw-mode full` — живой сценарий через Playwright MCP на `{{BASE_URL}}` (без `browser_close`).
- Каждый критерий приёмки должен быть фактически выполнен. «Готово» = все критерии + гейт зелёные.

## Шаг 4 — Финализация
- По умолчанию (commit-режим, у репо нет remote): закоммить логическими единицами на ветке. НЕ пушить, PR не открывать.
- Если настроен GitHub remote и режим `pr`: `git push -u origin <branch>` + `gh pr create` в `main` (тело: суть + тест-план + матрица из Шага 5). PR НЕ мержить.

## Шаг 5 — Сценарии для ручной проверки
- Собери список шагов, как руками проверить результат (UI или API), + матрицу ожиданий (вход → ожидаемый результат). Точные ярлыки UI/имена эндпоинтов. Это идёт в комментарий карточки.

## Правила
- Где нужен выбор и в карточке нет указания — бери разумный дефолт, не спрашивай (автономный режим).
- «Баг» = поведение, расходящееся с критериями приёмки или схемой данных. Косметику без необходимости не трогай.
- Минимальный диф. Не ломай существующие тесты и общие хелперы. На `main` не коммить.
