# Review Playbook — ahillary-will-do-abs

**Версия:** 2.0
**Дата:** 2026-06-21

Единственный источник процедуры по карточке — и для автопилота, и для ручной работы.
Проект: личный локальный трекер (FastAPI + SQLite + React/Vite/Tailwind), один пользователь.
Автопилот перечитывает этот файл на КАЖДОЙ карточке — правки подхватываются без перезапуска.

## Плейсхолдеры (подставляет оркестратор)
- `{{BASE_URL}}` — фронт (Vite), обычно `http://localhost:5173`
- `{{API_HEALTH_URL}}` — health бэкенда, обычно `http://localhost:8000/health`
- `{{CREDS}}` — вход в приложение: `SEED_USER_EMAIL` / `SEED_USER_PASSWORD` из `backend/.env`
- `{{BRANCH_NAME}}` — ветка `feat/…` или `fix/…`
- `{{TASK}}` / `{{ACCEPTANCE}}` — заголовок и блок «Критерии приёмки» карточки

## Карта флоу

```
        ┌────────────────────────────────────────────────────────────┐
        │ ШАГ 0 — Env-preflight                                        │
        │ /health == {"status":"ok"} + фронт отвечает (если нужно)    │
        └───────────────────────────────┬────────────────────────────┘
                                         │ окружение зелёное
                                         ▼
        ┌────────────────────────────────────────────────────────────┐
        │ ШАГ 1 — Разбор задачи + UI-сценарии                         │
        │ определить TASK_SURFACE; для UI-карточек — 3-5 сценариев    │
        └───────────────────────────────┬────────────────────────────┘
                                         ▼
        ┌────────────────────────────────────────────────────────────┐
        │ ШАГ 2 — Реализация (ветка от main, TDD, минимальный диф)    │
        └───────────────────────────────┬────────────────────────────┘
                                         ▼
        ┌────────────────────────────────────────────────────────────┐
        │ ШАГ 3 — Проверка                                             │◀────┐
        │ backend-only: ruff + pytest                                  │     │
        │ UI-карточка:  + Playwright (Claude сам кликает в браузере)   │     │
        │ выход: VERDICT (PASS/FAIL) + BUG_LIST                        │     │
        └───────────────────────────────┬────────────────────────────┘     │
                                         │                                  │
                       FAIL / есть баги  │           PASS / 0 багов         │
                                         ▼                                  │
        ┌────────────────────────────────────────────────────────────┐     │
        │ ШАГ 4 — Фиксы по BUG_LIST → возврат на Шаг 3               │─────┘
        │ (максимум 3 цикла; после 3 — стоп и FAILED)                 │
        └───────────────────────────────┬────────────────────────────┘
                                         │ VERDICT = PASS
                                         ▼
        ┌────────────────────────────────────────────────────────────┐
        │ ШАГ 5 — Финализация (коммит/PR) + финальные сценарии        │
        └──────────────────────────────────────────────────────────┘
```

**Условие выхода из цикла 3↔4:** все гейты зелёные + все сценарии = PASS.
Любой FAIL или непустой BUG_LIST → Шаг 4 → снова Шаг 3.

---

## ШАГ 0 — Env-preflight

**Цель:** убедиться в состоянии окружения до реализации.

**Когда применять:** раз в начале + повторно, если на Шаге 3 всплыл инфра-сбой.

**Процедура:**

1. Убедись, что дерево чистое и ты на свежем `main` (`git status`, `git pull --ff-only`).
2. Если карточка наблюдаема через HTTP/UI — проверь серверы:
   - `curl -s {{API_HEALTH_URL}}` → ожидаю `{"status":"ok"}`.
   - Фронт: `curl -s -o /dev/null -w "%{http_code}" {{BASE_URL}}` → 200.
3. Для чисто backend-карточек (модели, конфиг, парсеры, seed) серверы не нужны — переходи на Шаг 1.

**Формат вывода:**
```
ENV-PREFLIGHT:
- git: OK (main, clean)
- api_health: OK | FAIL — <ответ>
- frontend: OK | FAIL | N/A
ВЕРДИКТ: READY | NOT_READY
```
При `NOT_READY` — перечисли блокеры и не продолжай.

## ШАГ 1 — Разбор задачи + UI-сценарии

**Цель:** понять задачу, определить слой и — для UI-карточек — собрать сценарии проверки.

**Процедура:**

1. Прочитай карточку целиком: заголовок, описание, чеклист **«Критерии приёмки»** — это Definition of Done.

2. Объясни задачу в РОВНО 2 абзацах: о чём задача, что добавляет/улучшает, как влияет на проект.

3. Определи **TASK_SURFACE** — реальный слой задачи (смотри на код, а не только на заголовок):
   - `backend-only` — результат не виден в UI (модели, конфиг, seed, парсеры). UI-сценарии не нужны.
   - `backend+UI` — бэкенд + фронтовое отображение результата.
   - `UI-only` — только фронт.

4. Сверь формулировку с кодом: если задача называет компонент/поле/маршрут, которого нет или который значит другое — зафикируй расхождение; если оно меняет ЧТО строить — выбери разумный дефолт и задокументируй выбор (не спрашивай).

5. Для **UI-карточек** (`backend+UI` / `UI-only`) составь **3-5 пошаговых UI-сценариев**:

**ЖЁСТКИЕ ПРАВИЛА СЦЕНАРИЕВ:**
- Только через интерфейс. Нельзя: прямые URL вместо кликов, прямой SQL, обход UI.
  - ПЛОХО: «Перейдите на `/workouts`».
  - ХОРОШО: «Откройте {{BASE_URL}}, войдите как {{CREDS}}, нажмите "Тренировки" в меню».
- До каждого экрана — реальный кликабельный путь. Если пути нет — помечай экран как недостижимый через UI и выноси в «Известные ограничения».
- Каждый шаг = конкретное действие + конкретный видимый результат.
- Ярлыки и заголовки бери из реально отрендеренного UI (сверься с компонентами); не выдумывай.
- Привяжи каждый сценарий к критерию приёмки.
- Приложение однопользовательское — роли не нужны, актор всегда единственный пользователь.

**Формат сценариев:**
```
### S1 — <название>
- Цель: <что проверяем>
- Предусловие: <если нужно>
- Шаги:
  1. <действие> → <ожидаемый видимый результат>
  2. …
- Итоговый ожидаемый результат: <одна строка>
- Закрывает: <критерий ACCEPTANCE>
```

## ШАГ 2 — Реализация

**Цель:** реализовать ровно то, что в критериях приёмки.

**Правила:**
- Ветка `{{BRANCH_NAME}}` от свежего `main`. На `main` не коммить.
- TDD там, где он даёт сигнал: бэкенд-логику покрывай тестами (RED → GREEN); трудно-тестируемый UI (drag&drop, file-picker) гейтируй через `tsc --noEmit` + живой Шаг 3.
- YAGNI: реализуй ровно критерии приёмки, не больше.
- Минимальный диф: НЕ давай ruff/prettier/eslint трогать несвязанные строки. Если форматтер переписывает чужой код — откати файл и переналожи только свои правки.
- Малые связные файлы (≤800 строк), явная обработка ошибок на границах.
- Секреты только из `.env` (ключи Anthropic/Trello никогда не в коде).
- Для UI-карточек: сделай результат НАБЛЮДАЕМЫМ в браузере — если бэкенд отдаёт новое поле, отрисуй его в UI, иначе Шаг 3 нечем прогонять.
- Не ломай существующие тесты и общие хелперы.

**Качественные гейты (прогони всё применимое к карточке):**
- Backend: `cd backend && ruff check . && ruff format --check . && pytest -q`
- Frontend: `cd frontend && npx tsc --noEmit && npx eslint src --max-warnings 0`
- Если есть vitest-тесты по карточке: `npx vitest run`

До **3 циклов** фиксов на гейтах; если после 3 циклов всё ещё FAIL — стоп и `AUTOPILOT_RESULT: FAILED`.

## ШАГ 3 — Проверка

### 3а. Гейт качества (всегда)

Прогони гейты из Шага 2. Все должны быть зелёными.

### 3б. Playwright-прогон (только для UI-карточек в `--pw-mode full`)

**Прогони ВСЕ сценарии из Шага 1 через MCP Playwright на `{{BASE_URL}}`.**
Ты сам кликаешь по живому UI — не пересказывай код, а воспроизводи действия.

**Процедура для каждого сценария:**
1. Иди ровно по шагам: каждое действие — реальный клик/ввод через Playwright.
   `browser_navigate` только на стартовый `{{BASE_URL}}` или `/login`; дальше — клики, без прямых URL.
2. После каждого шага делай snapshot и сверяй фактический экран с ожидаемым ПОБУКВЕННО (ярлыки, заголовки, тексты).
3. Снимай console-сообщения (`browser_console_messages`) и network-запросы (`browser_network_requests`).
4. НИКОГДА не вызывай `browser_close` — это рвёт соединение с браузером.

**ЧЕК-ЛИСТ КРИТЕРИЕВ ПРОВАЛА (любой пункт = баг, строгость не занижать):**
- `[НЕТ_КНОПКИ]` — требуемого сценарием элемента нет на экране.
- `[ЯРЛЫК≠СЦЕНАРИЙ]` — элемент есть, но его текст не совпадает со сценарием.
- `[ЛАГ]` — зависание/спиннер дольше ~2с, дёрганый редирект.
- `[КОНТРОЛ_НЕ_ОТРАЖАЕТ_РЕЗУЛЬТАТ]` — после мутирующего действия его контрол не изменился (не исчез / не задизейблился / не сменил ярлык). Смена статуса где-то рядом (тост) НЕ засчитывается.
- `[КОНСОЛЬ]` — любой `console.error` / необъяснённый warning.
- `[NETWORK]` — любой запрос 4xx/5xx (кроме явно ожидаемых сценарием отказов).
- `[НЕТ_КЛИК_ПУТИ]` — до целевого экрана нет кликабельного пути; ввод URL руками запрещён.
- `[НЕОЧЕВИДНЫЙ_ПУТЬ]` — чтобы выполнить шаг, нужно угадывать или скроллить к неочевидному месту.
- `[ENV]` — `{{API_HEALTH_URL}}` не зелёный / серверы упали → вернись на Шаг 0.

**ПРАВИЛО СТРОГОСТИ:** PASS по сценарию — только при 100% совпадении всех шагов и отсутствии любого пункта чек-листа. Малейшее сомнение → это баг.

**Формат вывода Шага 3:**
```
## Прогон
| Сценарий | Вердикт |
| S1       | PASS / FAIL |
| …        |             |
ИТОГ: PASS (0 багов) | FAIL (<N> багов)

## BUG_LIST
- BUG-01:
    сценарий/шаг: S<k> шаг <n>
    тип: [...]
    severity: CRITICAL|HIGH|MEDIUM|LOW
    факт: <что увидел>
    ожидание: <что требовал сценарий>
    репро: <кратко шаги>
(если багов нет — "## BUG_LIST\n(пусто)")
```

## ШАГ 4 — Фиксы → возврат на Шаг 3

**Когда:** `ИТОГ: FAIL` / BUG_LIST непустой.

**Правила:**
- Для каждого бага определи класс: `КОД_ФИЧИ` | `СЦЕНАРИЙ` | `ОКРУЖЕНИЕ`.
  - `[ЯРЛЫК≠СЦЕНАРИЙ]` — как правило, правка самого сценария (привести к реальному UI), а не кода.
  - `[ENV]` — вернись к Шагу 0.
- Каждый фикс проверяй повторным запуском, а не рассуждением.
- Минимальный диф, без постороннего реформата.
- Прогони гейты качества после правок.
- Баг в соседнем коде (вне scope) → отдельный коммит `fix(<домен>): …`, в отчёте отметь «пред-существующий, вне scope».
- **Максимум 3 цикла 3↔4**; если после 3 циклов ИТОГ != PASS → стоп и `AUTOPILOT_RESULT: FAILED`.

После фиксов → **вернись на Шаг 3**.

---

## ШАГ 5 — Финализация + финальные сценарии

**Когда:** Шаг 3 = PASS по всем сценариям + все гейты зелёные.

### 5а. Финализация

- Закоммить логическими единицами на ветке (тип: feat/fix/docs/test). НЕ пушить и PR не открывать (если оркестратор не задал иное в «Адаптации»).

### 5б. Финальные сценарии тестировщику

Пересобери финальную версию сценариев из Шага 1 (они уже проверены на Шаге 3):
- Те же UI-only правила (без URL/SQL), маршрут кликами от `{{BASE_URL}}`.
- Все ярлыки/заголовки — точно как в реальном UI (после сверки на Шаге 3).
- Для каждого сценария добавь: «Как срабатывает» (1-2 строки механики) и «Ожидаемый результат».
- В конце — краткая матрица ожиданий (что / где / что должно быть видно).
- Раздел «Известные ограничения» — если есть экраны без клик-пути или что-то намеренно вне scope.

**Для `backend-only` карточек** вместо сценариев: короткий smoke-guide (как проверить через `curl` или API-клиент + что ожидать).

---

## Правила (всегда)

- Где нужен выбор и в карточке нет указания — бери разумный дефолт, не спрашивай (автономный режим).
- «Баг» = поведение, расходящееся с критериями приёмки, или пункт из чек-листа Шага 3.
- Стек: backend — Python (venv) + FastAPI + SQLModel + SQLite; frontend — npm + Vite + React + TypeScript + Tailwind. НЕ вводи pnpm/yarn.
- Прод-сборку (`npm run build`) не гоняй — для проверки фронта хватает `tsc --noEmit` + eslint.
- В Playwright MCP НИКОГДА не вызывай `browser_close`.
- На `main` не коммить.
