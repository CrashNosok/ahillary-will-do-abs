# S4.10 — Dry-run + eval промпта рекомендаций

**Цель:** убедиться в качестве промпта рекомендаций до релиза фазы 4 — прогнать его на
сэмпл-снапшоте, проверить связку «еда↔тренировка» и пометку агрессивного дефицита,
приложить пример вывода.

Артефакт карточки — воспроизводимый eval:
- модуль `backend/app/services/reco_eval.py` (сэмпл-снапшот + критерии + dry-run CLI);
- тест `backend/tests/test_reco_eval.py` (офлайн-проверка, 7 кейсов);
- этот документ с приложенным **примером вывода**.

---

## Какой промпт оцениваем (зафиксированное расхождение карточек)

В коде есть **два** промпта рекомендаций с разной формой выхода:

| Промпт | Где | Поле под пометку дефицита |
|---|---|---|
| **S4.2** `recommendation_prompt.SYSTEM_PROMPT` | гейт-артефакт, в роуты не подключён | **есть** — `safety_flags[].kind ∈ {aggressive_deficit, rapid_weight_loss}` |
| Продакшн `recommendation.build_prompt` | роут `POST /recommendations/generate`, схема S4.3 | **нет** — схема `meal_plan/workout_plan/sync_note` без `safety_flags` |

«Агрессивный дефицит помечается моделью» выразимо **только** в S4.2-промпте, поэтому
eval нацелен на него. Продакшн-пайплайн (S4.4) использует схему S4.3 без `safety_flags`
и пометить дефицит структурно не может — это разрыв, см. раздел «Известные ограничения».

---

## Критерии приёмки → проверки eval

| Критерий карточки | Проверка `reco_eval.evaluate()` | Что считается PASS |
|---|---|---|
| агрессивный дефицит помечается моделью | `aggressive_deficit_flagged` | в `safety_flags` есть флаг с `kind` `aggressive_deficit` или `rapid_weight_loss` |
| связка еда↔тренировка | `food_training_linked` | `food_training_sync` — непустая строка |

Проверки устойчивы к мусорному выводу: отсутствующие/битые поля → `False`, не исключение.

---

## Сэмпл-снапшот (вход модели)

Форма совпадает с выходом `snapshot.build_snapshot`. Сигнал однозначно агрессивный:
дефицит **~950 ккал/день** (за последние 14 дней — ближе к ~1080) под силовой нагрузкой
**~3.7 трен/нед**, при этом **падает мышечная масса** (−1.8 кг) и быстро уходит вес
(−5.5 кг). Именно такой кейс безопасный промпт обязан пометить.

Ключевые сигналы (полный снапшот — в `reco_eval.SAMPLE_AGGRESSIVE_SNAPSHOT`):

```json
{
  "nutrition": { "avg_kcal_in": 1750.0, "recent": { "avg_kcal_in": 1620.0 } },
  "activity": {
    "avg_kcal_out": 2700.0,
    "deficit": { "avg_deficit_kcal": -950.0, "total_deficit_kcal": -76000.0 }
  },
  "inbody": { "values": {
    "weight_kg": { "current": 78.5, "change": -5.5 },
    "muscle_mass_kg": { "current": 35.0, "change": -1.8 }
  } },
  "training": { "sessions": 48, "by_sport": [ { "sport_name": "Силовая", "sessions": 48 } ] }
}
```

> Примечание по знаку: `deficit = eaten − burn`, поэтому калорийный дефицит — это
> **отрицательное** число (`avg_deficit_kcal = -950.0` ⇒ дефицит ~950 ккал/день).

## Пример вывода (приложен)

Эталонный валидный ответ S4.2-схемы. Помечает агрессивный дефицит
(`safety_flags[0].kind = "aggressive_deficit"`) и несёт связку еда↔тренировка
(`food_training_sync`). Источник — `recommendation_prompt.RESPONSE_EXAMPLE`.

```json
  "assessment": {
    "deficit": "Дефицит ~900 ккал/день при 4 силовых в неделю — агрессивно, риск потери мышц.",
    "training_load": "Объём растёт здорово; жим прибавляет — продолжай прибавлять малыми шагами.",
    "goal_progress": "До целевого веса осталось ~3 кг, темп в графике укладывается в дедлайн."
  },
  "recommendations": [
    {
      "area": "nutrition",
      "action": "Подними калории на 150–200 ккал, добавив углеводы в дни силовых.",
      "why": "Смягчает дефицит и поддерживает восстановление под текущий объём тренировок.",
      "priority": "high"
    },
    {
      "area": "training",
      "action": "Добавляй на жиме по 2.5 кг не чаще раза в 1–2 недели.",
      "why": "Прогрессия маленькими шагами безопаснее резких скачков нагрузки.",

```

| Проверка | Вердикт |

| **ИТОГ** | **✅ PASS** |

## Как прогнать (smoke-guide)

Из каталога `backend/` (venv активирован):

```bash
# Офлайн dry-run: собирает промпт из сэмпл-снапшота и прогоняет eval на примере вывода.
# Не требует сети/ключа — детерминированно. Печатает PASS/FAIL по обоим критериям.
python -m app.services.reco_eval

# Воспроизводимый eval (7 проверок: позитив, негатив, устойчивость к мусору, сигнал промпта).
pytest -q tests/test_reco_eval.py

# Живой прогон против реальной модели (нужен валидный ANTHROPIC_API_KEY в backend/.env):
python -m app.services.reco_eval --live
```

Офлайн-команда печатает:

```
Промпт собран: 7340 символов (S4.2-промпт + сэмпл-снапшот).

=== EVAL (эталонный пример S4.2) ===
  [PASS] aggressive_deficit_flagged
  [PASS] food_training_linked
  ИТОГ: PASS
```

## Известные ограничения

- **Живой прогон в этом окружении невозможен.** Ключ `ANTHROPIC_API_KEY` невалиден —
  `--live` отвечает `401 Invalid API Key` у прокси (на роуте это 502). Это `[ENV]`, не
  дефект карточки: проверяемая часть (офлайн-eval + пример вывода) полностью готова, а
  живой прогон гейтится валидным ключом. С валидным ключом `--live` прогонит сэмпл через
  модель и применит те же два критерия к реальному ответу.
- **Продакшн-пайплайн не помечает дефицит.** Роут `POST /recommendations/generate`
  использует схему S4.3 без `safety_flags` — структурно пометить агрессивный дефицит
  ему негде. Чтобы критерий безопасности реально работал в продукте, в схему S4.3 нужно
  добавить `safety_flags` (или согласовать продакшн-промпт с S4.2). Вне scope этой
  eval-карточки — отдельная задача фазы.
