# Батч (отложен): голосовые вопросы недели → корректировка плана

> Статус: **запланировано, не реализовано.** Помечено в коде:
> `frontend/src/components/WeeklyReportPanel.tsx` — `// TODO(batch: voice-weekly-questions)`.

## Зачем

В конце недели, при формировании отчёта и плана на следующую неделю (кнопка «Получить отчёт» →
`WeeklyReportPanel`), задавать пользователю короткие вопросы о причинах пропусков и **принимать
ответы голосом**, затем учитывать их при генерации плана. Сейчас отчёт показывает сводку недели и
формирует план через существующий `/recommendations/generate`, но без диалога о причинах.

## Что входит в батч

1. **Вопросы недели.** Генерировать 1–3 вопроса по «дырам» недели, видимым из данных:
   - не вносил еду/активность/тренировки в N днях → «почему пропустил логирование?»;
   - тренировок мало / нет → «почему не тренировался?»;
   - вес/замеры/фото не внесены → «почему пропустил недельные замеры?».
   Источник «дыр» уже есть на фронте: `WeekFill`/`DayFlags` в `WeeklyReportPanel`.

2. **Запись ответов голосом.** В проекте сейчас нет аудио-инфры (ни `MediaRecorder`, ни загрузки
   аудио). Нужно:
   - компонент записи (`MediaRecorder` API: старт/стоп, индикатор, переслушать);
   - загрузка аудио на бэкенд (multipart, как `uploadBodyPhoto`/`uploadAchievementProof`);
   - транскрипция (Whisper-совместимый эндпоинт или внешний STT) → текст ответа.

3. **Учёт ответов в плане.** Передавать транскрипты ответов в снапшот для `recommendation`-генерации
   (расширить `app/services/snapshot.py` / промпт `docs/eval-reco-prompt.md`), чтобы план
   корректировался под названные причины.

4. **Хранение (вероятно нужна новая таблица).** Чтобы помнить «забранные» недели, заданные вопросы и
   ответы (и не спрашивать повторно): модель `WeeklyReport`
   (`week_start`, `medal_overall`, `claimed_at`, `answers_json`/ссылки на аудио) + миграция alembic.
   Тогда же стоит персистить «забранность» недели, которая сейчас эфемерна на фронте
   (`CalendarHeatmap` хранит `claimed` в React-состоянии, без сервера).
